# План апгрейда BitTrend

Документ фиксирует перечень улучшений и **рекомендуемый порядок** внедрения с учётом зависимостей между данными, скорингом, UI и инженерией. Детали методологии и источников — в `plan.md` (§8.5, §8.7, §8.9, §8.10).

---

## Полный список пунктов апгрейда

### Данные и надёжность

| ID | Пункт |
|----|--------|
| D1 | **Третий fallback ончейна (CoinGecko)** — добавить зависимость, модуль с proxy MVRV/NUPL/SOPR по §8.10; вызывать из `get_btc_onchain()`, когда Glassnode и LookIntoBitcoin не дали пригодных значений. |
| D2 | **Явный показ качества ончейна в UI** — вынести в интерфейс `confidence`, `source`, `source_score` из LTB (логика уже есть), по духу §8.9, чтобы «тишина» не считалась полноценными данными. |
| D3 | **Разнесённые интервалы обновления (§8.7)** — вместо одного TTL на весь кэш: чаще цена/funding/OI, реже макро/ETF/ончейн (отдельные TTL по ключам в `DataFetcher` или аналоге). |
| D4 | **Макро по плану §8.5** — при желании: CPI, S&P 500 (yfinance/FRED); сейчас в `macro.py` Fed Funds, 10Y, DXY (FRED `DTWEXBGS`, не Yahoo как в примере плана — при необходимости унифицировать формулировки с документом). |

### Скоринг и методология

| ID | Пункт |
|----|--------|
| S1 | **Интеграция идей §8.10** — proxy volatility/drawdown, rolling z по прокси, опциональный `composite_onchain`; согласование знака/весов с `BitTrendScorer` или отдельный блок «ончейн-composite» в UI. |
| S2 | **Калибровка по бэктесту** — ноутбук уровня `backtest_onchain_proxy.ipynb`, подбор порогов/весов перед «боевыми» настройками (обязательный шаг перед продакшен-весами). |
| S3 | **Дрейф и регрессия** — `detect_drift` из storage/LTB: использовать при алертах или снижении веса метрик при подозрении на поломку парсера. |

### Продукт и UX

| ID | Пункт |

## Логичная последовательность внедрения

Этапы ниже — **рекомендуемый порядок**; внутри этапа порядок пунктов тоже имеет смысл.

### Этап 1 — Инженерия и управляемость

1. **E1** — централизованные ретраи, rate limit, backoff (до расширения источников и частоты запросов).
2. **E2** — YAML (и при необходимости `.env` для секретов) для весов, порогов, аллокации.
3. **E3** — актуализация `README.md` по мере изменений кода и зависимостей.

### Этап 2 — Данные и кэш

4. **D3** — раздельные TTL для быстрых и медленных блоков данных.
5. **D1** — модуль CoinGecko + вызов из `get_btc_onchain()` как третий fallback.
6. **D4** — расширение_macro (CPI, S&P и согласование с текстом плана) — по приоритету продукта.

### Этап 3 — Прозрачность в UI

7. **D2** — отображение качества ончейна (`confidence`, `source`, `source_score`).

### Этап 4 — Скоринг и валидация

8. **S1** — интеграция §8.10 в скорер/UI (после стабильного D1 и конфига E2).
9. **S2** — бэктест и калибровка порогов/весов до фиксации продакшен-конфига.
10. **S3** — дрейф/регрессия в алертах и/или динамическое снижение весов.

### Этап 5 — Продукт и исполнение

11. **P1** — персистентная история сигналов (SQLite/CSV).
12. **P2** — график composite vs цена (после S1 и ряда из прокси/CoinGecko).
13. **P3** — реальное исполнение через биржу (ccxt/ключи) — **последним**, когда данные и скоринг предсказуемы; до этого сохранить явное логирование MVP.
